## 1. 개요 / 목적

2단계에서 생성/추천받은 레시피를, 사용자 인증(로그인/계정)을 통해 식별된 사용자의 프로필에 저장하여 나중에 다시 확인할 수 있도록 하는 기능을 구현한다.

이 단계의 목적은 "레시피를 한 번 보고 끝나는 것"이 아니라, 마음에 든 레시피를 사용자 개인 계정에 귀속시켜 보관해두고, 기기나 브라우저를 바꾸더라도 로그인만 하면 동일한 저장 목록에 접근할 수 있게 하여 서비스의 재사용 가치를 높이는 것이다.

## 2. 사용자 시나리오

1. 사용자가 서비스에 로그인(또는 회원가입 후 로그인)한다. 로그인하지 않은 사용자는 레시피 저장 기능을 사용할 수 없다.
2. 사용자가 2단계 결과 화면에서 레시피를 확인한다.
3. 마음에 드는 레시피가 있으면, 사용자는 해당 레시피 옆(또는 하단)의 "저장" 버튼을 클릭한다.
4. (비로그인 상태로 저장을 시도한 경우) 시스템은 로그인/회원가입을 유도하는 안내를 제공한다.
5. 시스템은 해당 레시피(요리명, 재료, 조리 순서, 예상 조리시간 등)를 현재 로그인한 사용자의 프로필에 저장 시각과 함께 저장한다.
6. 저장이 완료되면 사용자에게 저장되었음을 알리는 간단한 피드백(예: 버튼 상태 변경, 토스트 메시지)을 제공한다.
7. 사용자는 언제든 자신의 프로필 내 "저장한 레시피" 목록 화면으로 이동하여, 과거에 저장한 레시피들을 목록으로 확인할 수 있다.
8. 사용자는 저장 목록에서 특정 레시피를 선택하여 상세 내용(재료, 조리 순서 등)을 다시 볼 수 있다.
9. 사용자는 더 이상 필요 없는 저장된 레시피를 목록에서 삭제할 수 있다.
10. 사용자가 다른 기기/브라우저에서 동일 계정으로 로그인하면, 동일한 저장 목록에 접근할 수 있다.

## 3. 기능 요구사항 (Functional Requirements)

### FR-0. 사용자 인증
- FR-0.1. 사용자는 회원가입 및 로그인을 통해 계정을 생성/이용할 수 있어야 한다.
- FR-0.2. 레시피 저장, 저장 목록 조회, 저장된 레시피 삭제 등 프로필 관련 기능은 로그인한 사용자만 이용할 수 있다.
- FR-0.3. 비로그인 사용자가 저장 액션을 시도하면, 로그인/회원가입 화면(또는 안내)으로 유도한다.
- FR-0.4. 인증 방식(이메일/비밀번호, 소셜 로그인 등 구체적인 수단)은 본 PRD에서 확정하지 않으며, 기술 설계 단계에서 결정한다.

### FR-1. 레시피 저장
- FR-1.1. 로그인한 사용자는 2단계에서 생성된 레시피 결과 화면에서 저장 액션(예: "저장" 버튼)을 통해 레시피를 자신의 프로필에 저장할 수 있어야 한다.
- FR-1.2. 이미 저장된 레시피를 다시 저장하려는 경우(중복 저장) 어떻게 처리할지 판단 가능한 상태 표시(예: 이미 저장됨 표시)를 제공한다.
- FR-1.3. 저장 동작은 사용자가 별도 페이지 이동 없이 결과 화면에서 즉시 수행할 수 있어야 한다.

### FR-2. 저장 데이터 구성
저장되는 레시피 데이터에는 최소한 다음 항목이 포함되어야 한다.
- FR-2.1. 요리(메뉴) 이름
- FR-2.2. 사용된 재료 목록
- FR-2.3. 조리 순서
- FR-2.4. 예상 조리 시간
- FR-2.5. 저장 날짜/시각 (언제 저장했는지 식별 가능해야 함)
- FR-2.6. 저장한 사용자를 식별할 수 있는 정보(사용자 계정과의 연결 관계)

### FR-3. 프로필 내 저장 목록 조회
- FR-3.1. 로그인한 사용자는 자신의 프로필 내에서, 자신이 저장한 레시피들을 모아 볼 수 있는 별도의 목록 화면(예: "저장한 레시피")에 접근할 수 있어야 한다.
- FR-3.2. 목록에는 로그인한 사용자 본인이 저장한 레시피만 표시되어야 하며, 다른 사용자의 저장 목록은 조회할 수 없다.
- FR-3.3. 목록에서는 각 레시피의 요리 이름과 저장 날짜 등 핵심 정보를 한눈에 볼 수 있어야 한다.
- FR-3.4. 목록에서 특정 레시피를 선택하면 저장된 상세 내용(재료, 조리 순서, 예상 조리시간)을 확인할 수 있어야 한다.
- FR-3.5. 저장된 레시피가 하나도 없는 경우, 빈 상태(empty state)를 사용자에게 안내한다.

### FR-4. 저장된 레시피 삭제/관리
- FR-4.1. 사용자는 자신의 저장 목록에서 개별 레시피를 삭제할 수 있어야 한다.
- FR-4.2. 삭제 시 실수로 인한 삭제를 방지하기 위한 확인 절차(예: 삭제 확인 다이얼로그)를 두는 것을 권장한다.
- FR-4.3. (선택) 저장된 레시피가 많아질 경우를 대비한 정렬(예: 최신순) 기능은 필요 시 고려할 수 있으나 필수 요구사항은 아니다.

## 4. 비기능 요구사항 (Non-Functional Requirements)

- NFR-1 (데이터 지속성 및 소유권): 저장된 레시피는 사용자가 명시적으로 삭제하기 전까지, 서버 측에 해당 사용자 계정과 연결된 상태로 유지되어야 하며, 브라우저/기기 변경과 무관하게 로그인 시 동일하게 접근 가능해야 한다.
- NFR-2 (접근 제어/보안): 사용자는 본인이 저장한 레시피에만 접근(조회/삭제)할 수 있어야 하며, 다른 사용자의 저장 데이터에 접근할 수 없어야 한다.
- NFR-3 (사용성): 저장/삭제 액션은 최소한의 클릭/조작으로 수행할 수 있어야 하며, 결과(성공/실패)에 대한 피드백이 즉시 제공되어야 한다.
- NFR-4 (일관성): 저장된 레시피의 데이터 구조는 2단계에서 생성되는 레시피 데이터 구조(요리명, 재료, 조리순서, 예상 조리시간)와 일치해야 하며, 저장 후 다시 불러왔을 때 원본과 동일한 내용이 표시되어야 한다.

## 5. 범위 (Scope) — 이 단계에서 다루는 것

- 사용자 인증(회원가입/로그인) 및 로그인 상태 확인
- 레시피 결과 화면에서의 저장 액션(버튼) 구현 (로그인 사용자 대상)
- 저장 데이터 구성(요리명, 재료, 조리순서, 예상 조리시간, 저장일시, 사용자 연결 정보)
- 사용자 프로필 내 저장된 레시피 목록 조회 화면
- 저장된 레시피 상세 재조회
- 저장된 레시피 삭제 기능
- 본인 소유 데이터에 대한 접근 제어(다른 사용자의 저장 목록 비공개)

## 6. 범위 밖 (Out of Scope)

- 인증 수단의 구체적 구현 방식(이메일/비밀번호, 소셜 로그인 등)의 확정 — 요구사항 수준(로그인이 필요하다)까지만 정의하며, 구체 수단은 기술 설계 단계에서 결정
- 비밀번호 재설정, 이메일 인증 등 계정 관리 부가 기능의 세부 사항
- 여러 계정 간 저장 목록 공유, 협업 기능
- 저장된 레시피의 공유, 소셜 기능
- 저장된 레시피에 대한 사용자 메모/태그/카테고리 분류 등 부가 편집 기능
- 1단계(재료 인식), 2단계(레시피 생성) 자체의 기능 (→ 각각 PRD_step1.md, PRD_step2.md 참고)
- 1·2단계를 하나의 화면 흐름으로 통합하는 UI/UX 설계 (별도로 필요 시 추후 논의)

## 7. 성공 기준 / 완료 조건 (Definition of Done)

- 사용자는 회원가입/로그인을 통해 계정을 생성하고 로그인할 수 있다.
- 로그인한 사용자가 레시피 결과 화면에서 저장 버튼을 클릭하면, 해당 레시피(요리명/재료/조리순서/예상조리시간/저장일시)가 해당 사용자의 프로필에 저장된다.
- 비로그인 사용자가 저장을 시도하면 로그인/회원가입으로 안내된다.
- 사용자는 자신의 프로필 내 저장 목록 화면에서 지금까지 저장한 레시피들을 확인할 수 있으며, 다른 사용자의 저장 목록은 볼 수 없다.
- 저장 목록에서 특정 레시피를 선택하면 저장 당시의 상세 내용을 정확히 다시 볼 수 있다.
- 사용자는 저장된 레시피를 목록에서 삭제할 수 있고, 삭제 후 목록에서 즉시 사라진다.
- 저장된 레시피가 없을 때는 빈 상태 안내 문구가 표시된다.
- 동일 계정으로 다른 기기/브라우저에서 로그인해도 동일한 저장 목록에 접근할 수 있다.
