# PRD Step 2 — 레시피 생성

- 프로젝트명: 냉장고를 부탁해
- 문서 버전: v1.0
- 작성일: 2026-07-01
- 단계: 2/3 (전체 3단계 중 두 번째 — 재료 목록 기반 레시피 생성)
- 선행 단계: [PRD_step1.md](./PRD_step1.md) 에서 확정된 재료 목록을 입력으로 사용한다.

---

## 1. 개요 / 목적

1단계에서 확정된 재료 목록(텍스트)을 입력으로 받아, (비전 모델과는 별개의) 텍스트 생성용 AI 모델(LLM)을 통해 해당 재료로 만들 수 있는 요리 레시피를 생성/추천하는 기능을 구현한다.

이 단계의 목적은 "재료 목록 → 레시피"라는 두 번째 파이프라인 구간을 완성하는 것이며, 1단계의 이미지 인식 로직과는 독립적으로 개발/검증 가능하도록 설계한다(입력은 재료 목록 텍스트이면 되므로, 1단계 없이도 임의의 재료 목록으로 이 단계를 단독 테스트할 수 있어야 한다).

## 2. 사용자 시나리오

1. 사용자(또는 상위 시스템)가 확정된 재료 목록을 레시피 생성 기능에 전달한다.
2. 시스템이 해당 재료 목록을 LLM에 전달하여 레시피 생성을 요청한다.
3. 레시피 생성 중임을 알리는 로딩 상태가 표시된다.
4. LLM이 생성한 레시피(요리명, 재료, 조리 순서, 예상 조리시간 등)를 화면에 표시한다.
5. 사용자는 생성된 레시피 내용을 확인한다.
6. (LLM 응답 실패 또는 형식 오류 시) 사용자는 재생성을 요청할 수 있다.

## 3. 기능 요구사항 (Functional Requirements)

### FR-1. 레시피 생성 요청
- FR-1.1. 시스템은 확정된 재료 목록(텍스트 리스트)을 입력으로 받아 LLM에 레시피 생성을 요청한다.
- FR-1.2. 레시피 생성에는 텍스트 생성이 가능한 LLM을 사용하며, 이는 1단계에서 사용하는 비전 모델과 별도의 모델(또는 별도의 호출)로 처리한다.
- FR-1.3. 레시피 생성 요청 시 최소 1개 이상의 재료가 존재해야 하며, 재료 목록이 비어 있는 경우 사용자에게 안내 후 요청을 진행하지 않는다.

### FR-2. 레시피 결과 형식
생성된 레시피는 아래 항목을 포함하는 구조화된 형태로 반환/표시되어야 한다.
- FR-2.1. 요리(메뉴) 이름
- FR-2.2. 사용되는 재료 목록 (1단계에서 전달받은 재료 중 실제 사용되는 항목, 필요 시 추가로 필요한 기본 조미료 등도 별도 표기 가능)
- FR-2.3. 조리 순서 (단계별 설명)
- FR-2.4. 예상 조리 시간
- FR-2.5. (선택) 난이도, 인분 수 등 부가 정보 — 생성 가능한 경우 포함하되 필수 항목은 아님

### FR-3. 레시피 개수 및 재생성
- FR-3.1. 시스템은 1개 이상의 레시피 후보를 생성하여 사용자에게 제시할 수 있다. (정확히 몇 개를 제시할지는 구현 단계에서 결정하며, 본 PRD에서는 "1개 이상"이라는 요구사항 수준까지만 정의한다.)
- FR-3.2. 사용자는 마음에 드는 레시피가 없을 경우 재생성을 요청할 수 있다.

### FR-4. 오류 처리
- FR-4.1. LLM API 호출 실패(네트워크 오류, 타임아웃, 서버 오류 등) 시 사용자에게 실패를 안내하고 재시도할 수 있는 방법을 제공한다.
- FR-4.2. LLM 응답이 기대한 형식(레시피 구조)을 따르지 않는 경우, 이를 감지하여 재요청하거나 사용자에게 오류를 안내하는 처리 로직을 둔다.

## 4. 사용자 조건 조정 기능 — 범위 밖 명시

레시피 생성 시 사용자가 조건(예: 매운맛 제외, 조리시간 제한, 인분 수 지정, 특정 재료 제외 등)을 추가로 입력하여 결과를 조정하는 기능은 **현재 요구사항이 확정되지 않았으므로 이 단계에서는 범위 밖(Out of Scope)으로 명시**한다. 추후 별도 논의를 통해 요구사항이 확정되면 후속 단계에서 다룬다.

## 5. 비기능 요구사항 (Non-Functional Requirements)

- NFR-1 (성능): 레시피 생성 요청부터 결과 표시까지 소요되는 시간 동안 사용자가 진행 상태(로딩)를 인지할 수 있어야 한다.
- NFR-2 (일관성): 동일하거나 유사한 재료 목록에 대해 매번 지나치게 비현실적이거나 재료와 무관한 레시피가 생성되지 않도록, 프롬프트 설계 시 재료 목록을 명확히 반영하는 방식을 고려한다.
- NFR-3 (안정성): LLM 응답 지연, 오류, 형식 불일치 등 예외 상황에서도 애플리케이션이 중단되지 않아야 한다.

## 6. 범위 (Scope) — 이 단계에서 다루는 것

- 확정된 재료 목록을 입력받아 LLM에 전달하는 연동 로직
- LLM을 통한 레시피 생성 (요리명, 재료, 조리 순서, 예상 조리시간 포함)
- 레시피 생성 결과의 화면 표시
- 레시피 재생성 요청 처리
- LLM 호출 실패/형식 오류에 대한 기본적인 오류 처리

## 7. 범위 밖 (Out of Scope)

- 사용자가 매운맛 제외, 조리시간 제한, 인분 수 등 조건을 지정하여 레시피를 커스터마이징하는 기능 (요구사항 미확정)
- 1단계 이미지 인식 로직 자체 (선행 단계의 산출물인 재료 목록을 입력으로만 사용)
- 3단계에서 다룰 전체 서비스 통합 UI/UX
- 레시피 저장, 즐겨찾기, 평가/리뷰, 공유 등 부가 기능
- 특정 LLM 모델/벤더의 확정 (요구사항 수준까지만 정의)

## 8. 성공 기준 / 완료 조건 (Definition of Done)

- 임의의 재료 목록(텍스트)을 입력하면 LLM을 통해 요리명, 재료, 조리 순서, 예상 조리시간을 포함한 레시피가 생성되어 화면에 표시된다.
- LLM 호출 실패 시 사용자에게 오류가 안내되고, 재생성/재시도가 가능하다.
- 재료 목록이 비어 있는 등 잘못된 입력에 대해 요청 전 적절히 안내된다.
- 1단계 없이 임의의 재료 목록만으로도 레시피 생성 기능이 독립적으로 동작함을 확인할 수 있다.
