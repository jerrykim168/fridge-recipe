# PRD Step 1 — 이미지 입력 및 AI 기반 재료 인식

- 프로젝트명: 냉장고를 부탁해
- 문서 버전: v1.0
- 작성일: 2026-07-01
- 단계: 1/3 (전체 3단계 중 첫 번째 — 이미지 업로드 및 재료 인식)

---

## 1. 개요 / 목적

사용자가 냉장고 내부를 촬영한 사진을 업로드하면, AI 비전(이미지 인식) 모델이 사진 속 식재료를 분석하여 텍스트 형태의 재료 목록으로 추출하는 기능을 구현한다.

이 단계의 목적은 "사진 → 재료 목록"이라는 파이프라인의 입구를 안정적으로 완성하는 것이다. 이후 단계(레시피 생성, 전체 서비스 통합)는 이 단계에서 만들어진 재료 목록을 신뢰할 수 있는 입력값으로 사용하므로, 인식 실패나 저품질 인식에 대한 사용자 경험(재시도/수동 수정)을 함께 설계하는 것이 핵심이다.

## 2. 사용자 시나리오

1. 사용자가 웹 페이지에 접속하여 사진 업로드 영역을 확인한다.
2. 사용자가 냉장고 내부 사진 파일을 선택(또는 드래그 앤 드롭)하여 업로드한다.
3. 시스템이 업로드된 이미지를 서버로 전송하고, 업로드/분석 진행 상태를 화면에 표시한다.
4. AI 비전 모델이 이미지를 분석하여 인식된 재료 목록(예: 계란, 대파, 두부, 우유)을 반환한다.
5. 사용자는 화면에 표시된 재료 목록을 확인한다.
6. (인식 결과가 불완전하거나 틀린 경우) 사용자는 항목을 추가/삭제/수정하거나, 사진을 다시 업로드하여 재시도할 수 있다.
7. 사용자가 최종 재료 목록을 확정한다. (확정된 목록은 다음 단계인 레시피 생성의 입력으로 사용될 예정이나, 본 단계에서는 목록 확정 및 화면 표시까지를 범위로 한다.)

## 3. 기능 요구사항 (Functional Requirements)

### FR-1. 이미지 업로드
- FR-1.1. 사용자는 이미지 파일(JPEG, PNG 등 일반적인 사진 포맷)을 선택하여 업로드할 수 있어야 한다.
- FR-1.2. 업로드 방식은 파일 선택(클릭)과 드래그 앤 드롭 중 최소 하나 이상을 지원한다.
- FR-1.3. 업로드 가능한 파일 크기/포맷에 제한을 두고, 제한을 초과하거나 지원하지 않는 형식일 경우 사용자에게 명확한 안내 메시지를 표시한다.
- FR-1.4. 업로드한 이미지의 미리보기를 사용자에게 보여준다.

### FR-2. AI 비전 모델 연동
- FR-2.1. 업로드된 이미지를 비전 지원 AI 모델에 전달하여 이미지 속 식재료를 인식/추출한다.
- FR-2.2. 인식 결과는 재료명 리스트(텍스트) 형태로 반환되어야 하며, 각 항목은 화면에 개별적으로 표시 가능한 구조여야 한다.
- FR-2.3. 모델 호출 중 발생할 수 있는 지연을 고려하여, 분석 중임을 알리는 로딩 상태를 표시한다.
- FR-2.4. 모델 API 호출 실패(네트워크 오류, 타임아웃, 서버 오류 등) 시 오류를 감지하고 사용자에게 실패 사실과 재시도 옵션을 안내한다.

### FR-3. 인식 결과 확인 및 보정 (저정확도/실패 대응 UX)
- FR-3.1. 인식된 재료가 하나도 없거나(재료 미검출) 신뢰도가 낮다고 판단되는 경우, 사용자에게 "인식된 재료가 없습니다" 등 명확한 메시지를 보여준다.
- FR-3.2. 사용자는 인식 결과 목록에서 항목을 수동으로 삭제할 수 있어야 한다.
- FR-3.3. 사용자는 인식되지 않은 재료를 텍스트로 직접 추가할 수 있어야 한다.
- FR-3.4. 사용자는 동일 사진에 대해 재분석을 요청하거나, 새로운 사진으로 교체하여 다시 업로드(재시도)할 수 있어야 한다.
- FR-3.5. 재시도 횟수나 처리 방식에 특별한 제한이 필요한 경우, 남용 방지를 위한 최소한의 제한(예: 짧은 시간 내 반복 요청 제한)을 고려할 수 있다. (구체적 수치는 구현 단계에서 결정)

### FR-4. 결과 표시
- FR-4.1. 인식된 재료 목록은 업로드한 이미지와 함께 화면에 표시된다.
- FR-4.2. 사용자가 수정(추가/삭제)한 최종 재료 목록 상태가 화면에 즉시 반영된다.

## 4. 비기능 요구사항 (Non-Functional Requirements)

- NFR-1 (성능): 이미지 업로드부터 인식 결과 표시까지의 체감 대기 시간을 최소화하고, 처리 중에는 반드시 진행 상태(로딩 인디케이터 등)를 노출하여 사용자가 멈춘 것으로 오인하지 않도록 한다.
- NFR-2 (안정성): 비전 모델 API 호출 실패, 타임아웃 등 예외 상황에서도 애플리케이션이 중단되지 않고 사용자에게 오류를 안내해야 한다.
- NFR-3 (데이터 처리): 업로드된 이미지는 재료 인식이라는 목적 범위 내에서만 처리하며, 이미지의 저장/보관 정책(저장 여부, 보관 기간)은 개인정보 및 저장소 비용을 고려하여 이후 설계 단계에서 구체화한다.
- NFR-4 (호환성): 일반적인 데스크톱/모바일 브라우저에서 사진 업로드 기능이 정상 동작해야 한다. (구체적인 반응형 UI 요구사항은 3단계 통합 PRD에서 다룬다.)

## 5. 기술 고려사항

- 이미지 속 재료를 인식하기 위해서는 **비전(이미지 인식) 기능을 지원하는 AI 모델**이 필요하다. 현재 프로젝트에서 활용 가능한 LLM 연동 방식(OpenRouter 경유 DeepSeek 계열)은 텍스트 생성에 특화되어 있으며, DeepSeek 계열 모델은 비전/이미지 인식을 지원하지 않을 수 있다.
- 따라서 1단계(이미지 → 재료 인식)에는 **비전 지원 모델이 별도로 필요하다**는 점을 요구사항으로 명시한다. 구체적인 모델/벤더 선정은 본 PRD의 범위가 아니며, 추후 기술 검토 단계에서 결정한다.
- 2단계(레시피 생성)에서 사용할 LLM과 1단계에서 사용할 비전 모델은 서로 다른 모델일 수 있으며, 두 모델 간 연동(재료 목록을 텍스트로 전달)이 가능한 구조로 설계되어야 한다.

## 6. 범위 (Scope) — 이 단계에서 다루는 것

- 냉장고 사진 업로드 UI 및 업로드 처리
- 비전 AI 모델을 통한 이미지 → 재료 목록 인식 연동
- 인식 실패/저정확도 상황에 대한 사용자 경험(재시도, 수동 수정)
- 최종 확정된 재료 목록의 화면 표시

## 7. 범위 밖 (Out of Scope)

- 재료 목록을 이용한 레시피 생성 (→ 2단계에서 다룸)
- 전체 서비스 통합 UI/UX, 반응형 디자인 세부 사항 (→ 3단계에서 다룸)
- 회원가입/로그인, 결제, 소셜 공유 등 부가 기능
- 이미지 저장소 장기 보관, 사용자별 업로드 이력 관리
- 특정 비전 AI 모델/벤더의 확정 (요구사항 수준까지만 정의)

## 8. 성공 기준 / 완료 조건 (Definition of Done)

- 사용자가 이미지를 업로드하면 비전 AI 모델을 통해 재료 목록이 정상적으로 반환되고 화면에 표시된다.
- 재료가 인식되지 않거나 일부만 인식된 경우에도 사용자가 이를 인지하고, 수동으로 항목을 추가/삭제하거나 재시도할 수 있다.
- API 오류/타임아웃 등 예외 상황에서 사용자에게 오류 메시지가 명확히 노출되며 애플리케이션이 비정상 종료되지 않는다.
- 사용자가 최종적으로 확정한 재료 목록이 이후 단계(레시피 생성)에 전달할 수 있는 형태(텍스트 리스트)로 정리된다.
